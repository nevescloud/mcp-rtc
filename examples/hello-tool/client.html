mcp·rtc hello-tool · client Path A · deferred

Path A — in-browser inference calling get_greeting via mcp-rtc — is intentionally not built yet.

What it would do

Open this page in a second browser tab. It would load a small instruction-tuned model entirely client-side (default: Qwen2.5-0.5B-Instruct via WebLLM, with Chrome's built-in Gemini Nano as a fallback when the browser supports it). The page would connect to hello.html via mcp-rtc, list the tool, prompt the local model with "call get_greeting and tell me what the tab said," the model would issue a tool call, the call would route peer-to-peer, the model would speak the result.

No API key. No external service. No vendor in the loop. The path that would prove mcp-rtc is genuinely model-agnostic.

Why deferred

Reliable structured tool-calling at sub-1B parameters is fragile in May 2026. The honest options today:

  • Qwen2.5-0.5B via WebLLM with prompt-scaffolded tool format — small, fast first load, but tool-call output is inconsistent.
  • Llama-3.2-3B-Instruct via WebLLM — much more reliable but ~2 GB initial download. Hard sell as a demo.
  • Chrome's window.ai / Prompt API — built-in to Chrome 127+, but tool-calling support is limited.
  • Apple Foundation Model — Safari only on Apple Intelligence devices.

A flaky Path A becomes a liability rather than evidence — it makes someone glance at the demo, see the model misfire, and walk away thinking the rest probably doesn't work either. Worth waiting until the in-browser tool-calling story stabilizes (or until the specific demo can be made deterministic).

What would unblock it

  • A small-enough model (≤ 1B params) that reliably emits parseable tool calls for a single-tool, single-arg schema.
  • Or: a Chrome Prompt API release that exposes structured tool-use bindings on consumer hardware.
  • Or: a willingness to ship the Llama-3.2-3B demo with a "first run downloads ~2 GB" warning, accepting that as the cost of model-agnostic.

Working paths today

Path B — Claude Code via stdio bridge. Add @nevescloud/mcp-rtc-bridge to ~/.claude/settings.json and ask Claude. Works against hello.html and any mcp-rtc server.

Path C — Claude.ai / Desktop via WebMCP. Open neves.cloud/b/ with a site id in the fragment, in a Chrome 146+ tab with a WebMCP consumer attached (Anthropic Claude extension, hatch, etc.). Headline pattern; no install on the AI side.

Tracking
